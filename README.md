## 🩺 Who This Is For

This style guide is built for:

- **Technical writers** documenting DICOM, HL7/FHIR, PACS, and AI radiology software
- **Software engineers** writing their own docs, release notes, or API references
- **Product managers** writing feature descriptions, user stories, and changelogs
- **Teams adopting docs-as-code** who need a medical-domain-specific starting point

---

## 🚫 The Problem This Solves

General style guides are excellent — but they were not built for the unique language, workflows, and compliance context of medical imaging informatics software.

This guide fills that gap. It covers:

- How to write about clinical workflows without clinical ambiguity
- Terminology standards for DICOM, HL7, FHIR, PACS, and AI/ML radiology features
- How to write UI text, error messages, and tooltips for clinical software
- Release notes standards for regulated health IT products
- API documentation patterns for medical imaging endpoints

## 🚀 How to Use This Guide

### Option 1 — Browse on GitHub
Navigate the folders above and read the guides directly in your browser.

### Option 2 — Clone Locally

git clone https://github.com/CMI-Suman-Lata-Malik/docs-as-code-style-guide.git

Open in any Markdown editor (VS Code recommended).

### Option 3 — Use the Templates
Copy any file from the `/templates` folder directly into your project as a  
starting point for your own documentation.
